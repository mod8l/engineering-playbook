# Code Review

I treat code review as a conversation about correctness, maintainability, and shared understanding. The best reviews catch real problems, teach something small, and leave the author better off than when they opened the pull request.

## Guiding Principles

- **Review the change, not the person.** Suggest, do not accuse.
- **Distinguish blockers from preferences.** Use `must`, `should`, and `consider` explicitly.
- **Respond promptly.** A slow review is a bottleneck, not a quality gate.
- **Approve when good enough.** Perfect is the enemy of shipped.
